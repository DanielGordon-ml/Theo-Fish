# Dual-Graph Knowledge Graph Builder — Design Specification

**Date:** 2026-03-16
**Status:** Approved
**Supersedes:** 2026-03-15-knowledge-graph-builder-design.md

---

## 1. Overview

The Knowledge Graph Builder is being redesigned from a flat entity-extraction pipeline producing Obsidian markdown to a dual-graph architecture backed by Neo4j. The current implementation creates a star-graph topology where every entity connects back to an index.md file, producing trivial relationships. The new system extracts structured mathematical knowledge into two coupled graphs that capture both vocabulary and reasoning.

**Priority A (this spec):** Dual-graph architecture (D), intra-document dependency extraction (A), conditional/typed edges (C).
**Priority B (separate spec):** Cross-document concept alignment (B) — deferred until graph has enough nodes for meaningful reconciliation.
**Out of scope:** Fine-tuning data generation — acknowledged as a downstream consumer of the Neo4j schema, not designed here.

### Core architectural decisions

- **Neo4j 5.26-community** (Docker local) is the canonical source of truth. Connection config is parameterized for future Aura migration.
- **Obsidian vault** is a browsable mirror generated by an exporter, with scoped sync-back for human curation.
- **`_meta/schema.yaml`** drives extraction prompts, validation, and vault export. Schema changes are config changes, not code changes.
- **Two-pass extraction** with hybrid scoping: per-section concepts → paper-wide merge → per-section claims grounded against merged concepts.
- **Concept deduplication cascade**: embedding pre-filter → string normalization → LLM confirmation, each stage reducing work for the next.

## 2. Dual-Graph Data Model

### 2.1 Graph A — Concept Ontology

Nodes represent mathematical objects. The current 11 entity types are demoted from node type to a `rhetorical_origin` metadata field. Primary typing comes from a semantic schema.

**Node label:** `:Concept`
**Unique constraint:** `slug` (deterministic from canonical name via `slugify()`)

| Property | Type | Owner | Description |
|---|---|---|---|
| slug | String | pipeline | Deterministic from canonical name. Immutable after creation. |
| name | String | human | Canonical display name. Pipeline sets initial; human may refine. |
| semantic_type | String | pipeline | One of: structure, space, operator, property, quantity, relation, class |
| rhetorical_origin | String | pipeline | Original entity type: definition, notation, remark, example |
| canonical_definition | String | human | Curated definition. Pipeline sets initial; human may refine. |
| formal_spec | String | pipeline | Formal specification, e.g., "(S, A, P, R, γ)" |
| components | String[] | pipeline | Named subparts (for structures) |
| ts_domain | String | pipeline | Domain of operator (promoted from type-specific for queryability) |
| ts_codomain | String | pipeline | Codomain of operator (promoted) |
| ts_applies_to | String | pipeline | What a property applies to (promoted) |
| ts_member_type | String | pipeline | Member type for class (promoted) |
| ts_objective | String | pipeline | Objective for problem type (promoted) |
| type_specific_fields | Map/JSON | pipeline | Remaining semantic-type-specific fields from schema.yaml that are not promoted to top-level (e.g., dimensionality, topology, arity, linearity). Stored as a JSON map. The 5 `ts_*` fields above are promoted because they are the most frequently queried; all others stay in the JSON bag for flexibility. |
| aliases | String[] | shared | Pipeline appends new variants; human may curate. Union merge. |
| status | String | shared | canonical \| review_needed \| deprecated. Pipeline flags; human resolves. |
| embedding | Float[] | pipeline | Vector embedding for dedup similarity search |
| human_notes | String | human | Free-form human annotations. Pipeline never touches. |
| date_modified | String | pipeline | ISO timestamp of last modification. Updated on every write. |

**Semantic types** (defined in schema.yaml with required/optional fields):

| Type | What It Captures | Typical Source |
|---|---|---|
| structure | Named mathematical objects and constructions | definitions |
| space | Sets, domains, manifolds, topological spaces | definitions |
| operator | Functions, maps, transformations between spaces | definitions, notations |
| property | Attributes that structures or spaces can possess | definitions, remarks |
| quantity | Measurable or computable scalar/vector values | definitions, notations |
| relation | Named binary or n-ary relationships | definitions |
| class | Families or categories of structures | definitions, remarks |
| problem | Named problems or tasks to be solved | definitions, remarks |
| criterion | Objective functions, loss functions, optimality conditions | definitions |

> **Taxonomy evolution:** The initial taxonomy is deliberately broader than the minimum to avoid shoehorning (e.g., "multi-armed bandit problem" → `problem`, "expected regret minimization" → `criterion`, not forced into `structure` or `quantity`). Since the schema is config-driven, adding or splitting types is cheap. However, the initial set shapes extraction prompt quality, so covering the common cases matters.

### 2.2 Graph B — Reasoning / Claims Graph

Nodes represent assertions about the concepts in Graph A. The existing rhetorical types map naturally here.

**Node label:** `:Claim`
**Unique constraint:** `slug` (composed from `source_paper_slug` + `--` + `claim_slug`)

| Property | Type | Owner | Description |
|---|---|---|---|
| slug | String | pipeline | Composed from `source_paper_slug` + `--` + claim label slug. Immutable. |
| source_paper_slug | String | pipeline | Slug of the source paper. Used for composite index and vault file naming. |
| label | String | pipeline | Original label from paper: "Theorem 4.1", "Lemma 3.3" |
| name | String | pipeline | Same as `label` for claims. The schema.yaml `required_fields` reference `name` — for claims this is the label. |
| claim_type | String | pipeline | theorem, lemma, proposition, corollary, conjecture, algorithm |
| assumptions | String[] | pipeline | List of stated assumptions |
| conclusion | String | pipeline | The claim's conclusion statement (LaTeX preserved) |
| proof_technique | String | pipeline | Identified technique: induction, contradiction, construction, etc. |
| strength | String | pipeline | exact \| asymptotic \| approximate \| existential |
| statement | String | pipeline | Full statement text with LaTeX |
| proof | String | pipeline | Full proof text if available |
| section | String | pipeline | Source section heading |
| status | String | human | verified \| unverified \| disputed. Human curation flag. |
| human_notes | String | human | Free-form human annotations. Pipeline never touches. |
| date_modified | String | pipeline | ISO timestamp of last modification. |

**Claim types** (defined in schema.yaml with required/optional fields):

| Type | Role | Required Fields |
|---|---|---|
| theorem | Major proven result | name, assumptions, conclusion, proof_technique |
| lemma | Supporting proven result | name, assumptions, conclusion |
| proposition | Intermediate standalone result | name, assumptions, conclusion |
| corollary | Direct consequence of another claim | name, conclusion, derived_from |
| conjecture | Unproven claim | name, statement |
| algorithm | Procedural claim achieving specified behavior | name, input_spec, output_spec, guarantee |

### 2.3 Source Nodes

Paper provenance as first-class nodes.

**Node label:** `:Source`
**Unique constraint:** `arxiv_id`

| Property | Type | Owner |
|---|---|---|
| arxiv_id | String | pipeline |
| title | String | pipeline |
| authors | String[] | pipeline |
| date_published | String | pipeline |
| date_extracted | String | pipeline |
| human_notes | String | human |

### 2.4 Provenance Nodes

Per-paper formulations of a concept. Append-only — one Provenance node per (concept, source) pair.

**Node label:** `:Provenance`

| Property | Type | Owner | Description |
|---|---|---|---|
| concept_slug | String | pipeline | Slug of the linked Concept node. Part of uniqueness constraint. |
| source_arxiv_id | String | pipeline | ArXiv ID of the linked Source node. Part of uniqueness constraint. |
| formulation | String | pipeline | This paper's formulation of the concept |
| formal_spec | String | pipeline | This paper's formal specification |
| section | String | pipeline | Source section heading |
| extraction_confidence | Float | pipeline | Extraction confidence score |

Connected via:
- `(:Provenance)-[:FORMULATION_OF]->(:Concept)` — carries `match_method` and `match_confidence`
- `(:Provenance)-[:SOURCED_FROM]->(:Source)`

### 2.5 Relationship Types

**Concept → Concept (Graph A edges):**

| Relationship | Inverse | Properties |
|---|---|---|
| GENERALIZES | SPECIALIZES | condition: String |
| SPECIALIZES | GENERALIZES | restriction: String |
| HAS_COMPONENT | COMPONENT_OF | — |
| INSTANCE_OF | HAS_INSTANCE | parameters: String |
| DUAL_OF | DUAL_OF | duality_type: String |
| EQUIPPED_WITH | — | compatibility_condition: String |

**Claim → Claim (Graph B edges):**

| Relationship | Inverse | Properties |
|---|---|---|
| DEPENDS_ON | — | role: String (existence_guarantee \| bound \| construction \| base_case \| inductive_step), applied_to: String |
| ENABLES | — | nature: String (constructive \| existential \| computational \| relaxation) |
| STRENGTHENS | WEAKENS | dimension: String (tighter_bound \| weaker_assumptions \| broader_scope) |
| WEAKENS | STRENGTHENS | dimension: String |
| CONTRADICTS | CONTRADICTS | resolution: String |
| EQUIVALENT_TO | EQUIVALENT_TO | under_assumptions: String |

**Inverse edge materialization:** Inverse edges (SPECIALIZES, COMPONENT_OF, HAS_INSTANCE) are NOT stored as separate Neo4j relationships. Only the primary direction is created. The inverse is a logical traversal — queries use `MATCH (a)-[:GENERALIZES]->(b)` to find what `b` specializes. The schema.yaml `inverse` field is metadata for prompt generation and validation (ensuring if A GENERALIZES B exists, no contradictory B GENERALIZES A exists), not a directive to create a second edge.

**Coupling edges (Claim → Concept):**

| Relationship | Properties |
|---|---|
| ABOUT | role: String (primary \| scope \| mechanism \| bound_target \| construction_input), aspect: String |

**Source attribution edges:**

These edges connect nodes to their paper provenance. `SOURCED_FROM` can appear on both Claim and Provenance nodes. Concepts do NOT have direct SOURCED_FROM edges — their provenance is always mediated through Provenance nodes (one per concept-source pair).

| Relationship | Pattern | Properties |
|---|---|---|
| SOURCED_FROM | Claim → Source | section: String, page: String, confidence: Float |
| SOURCED_FROM | Provenance → Source | section: String, confidence: Float |
| FORMULATION_OF | Provenance → Concept | match_method: String, match_confidence: Float |

### 2.6 Neo4j Constraints & Indexes

```cypher
// Uniqueness constraints
CREATE CONSTRAINT concept_slug FOR (c:Concept) REQUIRE c.slug IS UNIQUE;
CREATE CONSTRAINT source_arxiv  FOR (s:Source)  REQUIRE s.arxiv_id IS UNIQUE;
CREATE CONSTRAINT claim_slug    FOR (cl:Claim)  REQUIRE cl.slug IS UNIQUE;

// Provenance uniqueness — one provenance node per (concept, source) pair
// Keyed on concept_slug + source_arxiv_id properties stored on the Provenance node
CREATE CONSTRAINT provenance_unique FOR (p:Provenance)
  REQUIRE (p.concept_slug, p.source_arxiv_id) IS UNIQUE;

// Composite index for claim lookup by paper
CREATE INDEX claim_paper FOR (cl:Claim) ON (cl.source_paper_slug, cl.label);

// Full-text index for concept name/alias/formal_spec search
CREATE FULLTEXT INDEX concept_search FOR (c:Concept) ON EACH [c.name, c.formal_spec, c.aliases];

// Vector index for embedding-based dedup
// Dimensionality must match the chosen embedding model (see config)
CREATE VECTOR INDEX concept_embedding FOR (c:Concept) ON c.embedding
  OPTIONS {indexConfig: {
    `vector.dimensions`: 1536,
    `vector.similarity_function`: 'cosine'
  }};

// Status indexes
CREATE INDEX concept_status FOR (c:Concept) ON (c.status);
CREATE INDEX claim_status   FOR (cl:Claim)  ON (cl.status);
```

> **Embedding model:** The vector index dimensionality (1536) must match the embedding model output. The specific model is configured via `EMBEDDING_MODEL` in the environment. Default: OpenAI `text-embedding-3-small` (1536 dimensions). If changed to a model with different dimensionality, update the vector index accordingly. The embedding model is separate from the DeepSeek extraction model.
>
> **Benchmark before committing:** Mathematical concept dedup is specialized — "Q-function" and "action-value function" must land close together, but "value function" and "Q-function" must remain distinguishable. Before production use, benchmark the chosen model against 20-30 known synonym pairs from the domain. If the general-purpose model produces too many false positives in the 0.6–0.85 band, consider math-specialized embeddings or a fine-tuned adapter. The similarity thresholds (0.85 auto-resolve, 0.6 flag) may need model-specific calibration.

## 3. Field Ownership Model

Every node property is owned by exactly one of three categories. This determines write behavior for both the extraction pipeline and the Obsidian sync-back.

### Pipeline-owned
Pipeline writes freely. Exporter overwrites in Obsidian on every run. Sync-back ignores changes to these fields.

**Fields:** slug, semantic_type, rhetorical_origin, formal_spec, components, claim_type, assumptions, conclusion, proof_technique, strength, statement, proof, section, embedding, arxiv_id, title, authors, date_published, date_extracted, formulation, extraction_confidence

### Human-owned
Pipeline sets initial value on creation only (never overwrites). Sync-back reads these from Obsidian and writes to Neo4j.

**Fields:** name (on Concept), canonical_definition, human_notes, status (on Claim)

### Shared (append-only)
Pipeline appends new values. Human may add or remove. Sync-back merges both directions — union, never subtract.

**Fields:** aliases, status (on Concept — pipeline flags review_needed, human resolves)

> **Note:** The exhaustive list of fields per ownership category is in `_meta/schema.yaml` (Section 4, `field_ownership`). The lists above are summaries of the key fields.

### 3.1 Per-label field ownership disambiguation

The `status` field has different ownership depending on the node label:

| Field | :Concept | :Claim | :Source |
|---|---|---|---|
| status | **shared** (pipeline flags review_needed, human resolves to canonical/deprecated) | **human** (verified/unverified/disputed) | N/A |
| name | **human** (pipeline sets initial, human refines) | **pipeline** (same as label) | N/A |
| human_notes | **human** | **human** | **human** |

The field ownership implementation must dispatch on node label, not just field name.

### 3.2 Migration from v1

The existing vault at `data_vault/` contains v1 output (flat entity files organized per-paper). The migration path:
1. The v1 vault is archived as-is (no automatic migration of files to the new structure)
2. The new system creates a fresh vault structure (concepts/, claims/, sources/, _meta/)
3. All papers are re-extracted from `Data/cleaned/*.json` using the new pipeline, populating Neo4j from scratch
4. The v1 `graph_builder/` module is preserved as `graph_builder_v1/` until the new system is validated against the golden-paper test, then removed

## 5. Extraction Pipeline

### 5.1 Pipeline flow for a single paper

**Step 0 — Load & Prepare**
- Read cleaned JSON from `Data/cleaned/{arxiv_id}.json`
- Split into sections (reuse existing section_splitter: TOC strip, H1/H2 split, oversized handling)
- Load `_meta/schema.yaml` → parse concept_types, claim_types, edges
- Query Neo4j for existing concept slugs + embeddings (for dedup context)

**Step 1 — Pass 1: Concept Extraction (per-section, parallel)**
For each section (semaphore-limited concurrency):
- System prompt generated from schema.yaml: valid concept types + descriptions + required/optional fields + valid concept edge types
- User prompt: section text + top 10-20 embedding-similar existing concepts from Neo4j (pre-filtered) + instruction "If a concept matches an existing one, use that exact name"
- Response format: `{"concepts": [...], "concept_edges": [...]}`

**Step 2 — Concept Merge & Dedup (paper-wide, no LLM except targeted confirms)**

Step 2a — Intra-paper dedup: Collect all concepts from all sections → deduplicate by normalized name. Keep richest formulation (longest formal_spec, most components).

Step 2b — Cross-vault dedup cascade:
1. Compute embedding for each new concept
2. Vector similarity search against Neo4j `concept_embedding` index
3. String normalization (lowercase, expand known abbreviations like MDP → Markov Decision Process, strip articles)
4. Similarity > 0.85 → auto-resolve to existing slug
5. Similarity 0.6–0.85 → single targeted LLM confirmation call: "Is [new name] the same concept as [existing name]? Context: [both definitions]."
6. Similarity < 0.6 → new concept

Confidence and method propagate to Provenance nodes: `match_method` ("alias" | "embedding" | "embedding+llm" | "new"), `match_confidence` (1.0 for alias, cosine score for embedding, etc.). This enables downstream prioritization of human review and fine-tuning data weighting.

Output: Resolved concept list with slug mappings (new vs existing).

**Slugify algorithm** (deterministic, used for all slug generation):
1. Strip LaTeX commands: `$\epsilon$-greedy` → `epsilon-greedy`
2. Unicode NFKD normalization, strip combining marks
3. Lowercase
4. Replace non-alphanumeric characters with hyphens
5. Collapse consecutive hyphens to single hyphen
6. Strip leading/trailing hyphens
7. Example: "Markov Decision Process" → `markov-decision-process`
8. Example: "KL Divergence (D_KL)" → `kl-divergence-d-kl`

> **Slug stability guarantee:** The slugify algorithm is the identity anchor for the entire system. Slugs are used as Neo4j unique constraints, vault filenames, and wikilink targets. **The algorithm is frozen after first production run.** Any future changes to slugify (e.g., improved LaTeX stripping) require a migration script that updates all existing slugs in Neo4j, renames all vault files, and updates all wikilink references. This cost is intentional — it prevents silent orphaning of nodes at paper #200.

**Step 3 — Pass 2: Claim Extraction (per-section, parallel)**
For each section (semaphore-limited concurrency):
- System prompt generated from schema.yaml: valid claim types + descriptions + required/optional fields + valid claim edge types + coupling edge types (ABOUT)
- User prompt: section text + **section-local concepts** (concepts extracted from this section in Pass 1) + their **immediate Graph A neighbors from Neo4j** (e.g., if this section's "policy" GENERALIZES "stochastic policy" in the vault, include both). This keeps grounding precise without bloating context on 80+-concept papers.
- Response format: `{"claims": [...], "claim_edges": [...]}`

**Step 4 — Pass 3: Edge Enrichment (two-stage)**

Stage 3a — Per-section claim enrichment (parallel): For each section that produced claims, one LLM call with:
- The section text + all claims from that section (statements + proofs)
- The section-local concepts + the paper-wide concept list as reference
- Instructions to extract DEPENDS_ON, ABOUT, ENABLES edges with full conditional attributes (role, condition, technique, strength)
- **Batching claims by section** (not per-claim) keeps attribution reliable while cutting call count by 3-5x. A section with 5 claims is enriched in a single call since they share local context.

Stage 3b — Paper-wide gap-fill (single call): One call with:
- All claim **conclusions** (not full proofs) and all concept **formal_specs** (not full definitions)
- Already-extracted edges from Stage 3a (to avoid duplication)
- Instructions to find only cross-section implicit dependencies that Stage 3a missed (e.g., theorem in Section 5 depending on definition in Section 2 without explicit reference)
- Including conclusions and formal_specs (not just names/slugs) gives the LLM enough semantic content to detect implicit dependencies while staying well under context limits

**Step 5 — Validate & Write to Neo4j**

Validate against schema.yaml:
- Every claim has ≥1 ABOUT edge (error)
- Every concept has ≥1 Provenance node linked via FORMULATION_OF (error)
- Corollary has ≥1 DEPENDS_ON edge (warning)
- No unknown edge types (error)
- No unknown attributes on edges (warning)
- DEPENDS_ON edges don't form cycles (warning — may indicate equivalence, see cycle handling below)
- Generalizes/specializes inverse consistency (error)
- Log warnings/errors per severity defined in schema

Transaction strategy — granular by node type, batch by edges:
- Each valid concept is its own transaction (MERGE by slug, safe-merge semantics: set pipeline-owned fields, skip human-owned if node exists)
- Each valid claim is its own transaction (CREATE with SOURCED_FROM edge)
- Provenance nodes created individually (always append, link FORMULATION_OF → Concept + SOURCED_FROM → Source)
- All edges for the paper written in a single batch transaction (if any edge references a node that failed validation, that edge is excluded from the batch, not the whole batch)
- Concepts with divergent formulations (new provenance differs from canonical) flagged as `status: review_needed`

Failure modes — validation:
- 1 out of 20 claims fails validation → 19 valid claims written, failed claim logged with validation errors
- Valid concepts always contribute to the vault (other papers may need them)
- Edge batch never creates dangling references

Failure modes — LLM calls:
- If a Pass 1 section call fails after retries → that section's concepts are missing; pipeline proceeds with concepts from remaining sections. Log warning with section index.
- If a Pass 2 section call fails → claims from that section are missing; pipeline proceeds. Log warning.
- If a Pass 3a per-claim call fails → that claim has no enriched edges; it keeps whatever edges Pass 2 produced. Log warning.
- If Pass 3b (gap-fill) fails → cross-section implicit edges are missing; not fatal since per-claim edges from 3a cover explicit references. Log warning.
- If >50% of section calls fail for a paper → abort the paper, log as failed in BatchBuildResult. Don't write partial results that would misrepresent the paper's content.
- Dedup LLM confirmation failures → treat as "no match" (conservative — creates a new concept rather than incorrectly merging).

DEPENDS_ON cycle handling:
- Mathematical papers sometimes have mutually equivalent characterizations (Theorem A ⇔ Theorem B). These should use `EQUIVALENT_TO` edges, not bidirectional `DEPENDS_ON`.
- The extraction prompt explicitly instructs DeepSeek to use EQUIVALENT_TO for equivalences.
- If DEPENDS_ON cycles are detected during validation, they are flagged as **warnings** (not errors), logged with the cycle path, and the edges are still written. The cycle may indicate an extraction error or a genuine equivalence that should be reviewed.

### 5.2 Schema-driven prompt generation

The prompt generator reads `schema.yaml` at runtime and produces LLM prompts dynamically. Adding a new concept type or edge type requires editing `schema.yaml` only — the prompt generator includes it automatically.

Input: schema.yaml section (concept_types or claim_types) → Output: formatted prompt section listing valid types, required/optional fields, and edge types with descriptions and attributes.

### 5.3 LLM call budget per paper

| Pass | Calls | Parallelism | Context Size |
|---|---|---|---|
| Pass 1 — Concepts | N (one per section) | Parallel (semaphore) | Section text + schema excerpt + 10-20 existing concepts |
| Dedup confirmations | 0–5 (only for 0.6–0.85 similarity) | Parallel | Two definitions only (very small) |
| Pass 2 — Claims | N (one per section) | Parallel (semaphore) | Section text + schema excerpt + section-local concepts + neighbors |
| Pass 3a — Section enrichment | S (one per section with claims, S ≤ N) | Parallel (semaphore) | Section text + section claims + concepts |
| Pass 3b — Gap-fill | 1 | Sequential | All claim conclusions + concept formal_specs + existing edges |
| **Total** | **2N + S + 1 + 0–5 dedup** | | For a 10-section paper: ~21–26 calls (vs per-claim: ~36–41) |

Token cost roughly 3x current pipeline due to the second extraction pass + per-claim edge enrichment. Embedding computation is separate (local or API).

**Concurrency:** Default semaphore limit is 10 (matching v1's `DEFAULT_CONCURRENCY`). Configurable via `--concurrency` CLI flag. Pass 3a scales with claim count (M calls), which can be 30+ for dense papers — the semaphore prevents overwhelming the DeepSeek API.

## 6. Obsidian Vault — Export & Sync-Back

### 6.1 Vault structure

```
data_vault/
├── concepts/                          # Graph A — shared vocabulary
│   ├── markov-decision-process.md
│   ├── nash-equilibrium.md
│   └── ...
├── claims/                            # Graph B — organized by claim type
│   ├── theorems/
│   │   ├── intro-multi-armed-bandits--theorem-1-1.md
│   │   └── ...
│   ├── lemmas/
│   │   ├── intro-multi-armed-bandits--lemma-1-3.md
│   │   └── ...
│   ├── propositions/
│   ├── corollaries/
│   ├── conjectures/
│   └── algorithms/
├── sources/                           # Paper provenance nodes
│   ├── 1904-07272.md
│   └── ...
├── _review/                           # Human curation queue
│   ├── concepts/                      # status: review_needed concepts
│   └── claims/                        # Claims with validation warnings
└── _meta/
    ├── schema.yaml                    # Single source of truth
    └── export-manifest.json           # Audit trail for every export run
```

The `_review/` folder is populated by the exporter when validation flags issues. Each file in `_review/` is a symlink (or copy with `review_issue` frontmatter field) to the corresponding concept or claim in the main vault, with the specific issue noted. When a human resolves the issue and syncs back, the exporter removes the `_review/` entry on the next run. This gives curators a natural workflow: process the review queue, sync-back, done.

Claims use flat files within type folders: `claims/{claim_type}/{paper-slug}--{claim-slug}.md`. Paper slug as filename prefix enables by-paper filtering; folder structure enables by-type browsing. Source files link to all their claims for paper-centric navigation.

### 6.2 Frontmatter schema

Every edge type gets two frontmatter keys:
- **Flat wikilink key** (e.g., `depends_on: ["[[concepts/brouwer-fixed-point-theorem]]"]`) — Obsidian resolves these for graph view and backlinks
- **Companion `_meta` key** (e.g., `depends_on_meta:`) — structured attributes (role, condition, context)

The pipeline writes both keys atomically from the same data, so they cannot drift apart. The body contains only mathematical content with natural prose wikilinks — all machine-readable edge data lives exclusively in frontmatter.

### 6.3 Export: Neo4j → Obsidian

**Trigger:** CLI command `--export` or automatically after pipeline run.
**Mode:** Full regeneration — lossy, re-runnable at will.

Process:
1. Query all :Concept nodes → write `concepts/*.md`
2. Query all :Claim nodes grouped by type → write `claims/{claim_type}/*.md`
3. Query all :Source nodes → write `sources/*.md`
4. Copy `schema.yaml` to `_meta/`
5. Write `_meta/export-manifest.json`

Atomicity:
- Write to a staging directory first
- Atomic move per file to the vault
- If git tracks the vault and human-owned files have uncommitted changes, abort with warning — prevents clobbering in-progress human edits

Per-file field handling:
- Pipeline-owned fields: overwrite always
- Human-owned fields: read current file, extract human-owned values, write new file preserving them. If file doesn't exist, use pipeline defaults.
- Shared fields: union merge (never subtract)

### 6.4 Sync-Back: Obsidian → Neo4j

**Trigger:** CLI command `--sync-back` (manual, never automatic).
**Mode:** Scoped to human-owned and shared fields only.

Human-owned fields synced:
- `name` (on Concept) — canonical display name
- `canonical_definition` (on Concept)
- `human_notes` (on any node)
- `status` (on Claim) — verified / unverified / disputed

Shared fields synced:
- `aliases` — union of Obsidian + Neo4j values
- `status` (on Concept) — human can resolve review_needed → canonical

Pipeline-owned fields: **IGNORED** — never read from Obsidian.

Safety:
- Dry-run mode (`--dry-run`) shows diff before writing to Neo4j
- Each change logged with timestamp and source
- Only processes files modified after last `export-manifest.json` timestamp

### 6.5 Export manifest — `_meta/export-manifest.json`

Written on every export run. Records:
- Timestamp of export
- Neo4j transaction ID or last-processed commit
- Count of files written / skipped / merged
- Human-owned fields that were preserved (which files, which fields)

Used by sync-back as reference point: only process files modified after the last export timestamp.

### 6.6 Idempotency guarantees

| Scenario | Behavior |
|---|---|
| New concept in Neo4j, no file in vault | Create file with pipeline defaults |
| Concept updated in Neo4j, file exists | Overwrite pipeline-owned, preserve human-owned |
| Human edited name/canonical_definition in Obsidian | Preserved on re-export (read before write) |
| Human added alias in Obsidian | Union-merged — human alias kept, pipeline aliases added |
| Concept in vault but deleted from Neo4j | File removed (full regen is authoritative) |
| No changes since last export | Files rewritten identically (idempotent) |

## 8. Testing Strategy

### 8.1 Principles

- **Test before code** per CLAUDE.md: test file written before implementation
- **Real Neo4j for integration**: Docker testcontainer (neo4j:5.26-community). No mocking the graph database.
- **Mocked LLM at unit boundary**: Unit tests mock the LLM client. Integration tests use recorded fixtures (VCR-style).
- **Schema as test fixture**: Tests load the real schema.yaml, not a test copy. Schema changes that break tests are a signal.

### 8.2 Test infrastructure

- **Neo4j testcontainer**: Shared fixture creates constraints + indexes on startup. Each test gets a clean database (`MATCH (n) DETACH DELETE n` between tests). Same image as production.
- **LLM fixtures**: Recorded responses in `tests/fixtures/llm/` for reproducible extraction tests.
- **Graph fixtures**: Handcrafted Neo4j state in `tests/fixtures/graph/` for dedup, export, and sync-back tests.
- **Pytest markers**: `@pytest.mark.unit` (no external deps), `@pytest.mark.integration` (requires Neo4j), `@pytest.mark.e2e` (full pipeline), `@pytest.mark.slow` (>10s). CI runs unit always, integration on PR, e2e on merge to main.

## 9. Interface Points for Future Work

### 9.1 Fine-tuning data generation (Priority B — separate project)

The Neo4j schema is designed to support downstream training data generation. The fine-tuning generator will consume:
- `:Concept` and `:Claim` nodes with their full properties
- Typed edges with attributes (DEPENDS_ON with role, ABOUT with aspect, etc.)
- The dual-graph structure for subgraph sampling (concept walks, reasoning chains)
- `match_confidence` on Provenance nodes for weighting reliability

The generator should be designed after running the extraction pipeline on 10-20 papers and inspecting the actual graph topology at scale.

### 9.2 Cross-document claim relationships

Two papers proving the same theorem create two :Claim nodes (correct, since proofs and formulations differ). But the ABOUT edges from both claims point to the same :Concept nodes. A future `RESTATES` edge ("same result, different formulation") and `REPROVES` edge ("same result, different proof") between cross-paper claims would enable powerful queries like "all proofs of concept X" and "which proof techniques are used for theorem class Y." These edge types are reserved in the schema (commented out) to prevent topology assumptions that would make them hard to add later.

### 9.3 Cross-document concept alignment (Priority B)

The dedup cascade handles per-paper concept matching during extraction. A future reconciliation pass can:
- Query Neo4j for `status: review_needed` concepts
- Batch ambiguous pairs into focused LLM calls: "Is [concept A from paper X] the same as [concept B from paper Y]?"
- Merge confirmed duplicates by updating FORMULATION_OF edges
- Use the provenance-weighted confidence scores to prioritize review

### 9.4 Neo4j → Aura migration

Connection config is parameterized (`NEO4J_BOLT_URI`, `NEO4J_AUTH`). Migration to Aura is a config change:
```env
NEO4J_BOLT_URI=neo4j+s://xxxx.databases.neo4j.io
NEO4J_AUTH=neo4j/password
```

Cypher and schema are identical between Community and Aura.
