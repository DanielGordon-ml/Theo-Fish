Extraction Pipeline — Two-Pass with Schema-Driven Prompts

Hybrid extraction: per-section concepts → paper-wide merge → per-section claims grounded against merged concepts

Pipeline Flow — Single Paper
0
Load & Prepare
Read cleaned JSON → split into sections (reuse existing section_splitter)
Load _meta/schema.yaml → parse concept_types, claim_types, edges
Query Neo4j for existing concept slugs + embeddings (for dedup context)
1
Pass 1 — Concept Extraction (per-section, parallel)
For each section (semaphore-limited concurrency):
System prompt (generated from schema.yaml):
• Valid concept types + descriptions + required/optional fields
• Valid concept edge types + descriptions
• Instructions: extract objects, not claims

User prompt:
• Section text
• Top 10-20 embedding-similar existing concepts from vault (pre-filtered)
• "If a concept matches an existing one, use that exact name"

Response format: JSON
{"concepts": [{"name": "...", "semantic_type": "...", "formal_spec": "...", ...}], "concept_edges": [{"source": "...", "target": "...", "type": "GENERALIZES", ...}]}
2
Concept Merge & Dedup (paper-wide, no LLM)
Step 2a — Intra-paper dedup:
Collect all concepts from all sections → deduplicate by normalized name
Keep richest formulation (longest formal_spec, most components)

Step 2b — Cross-vault dedup cascade:
1. Compute embedding for each new concept
2. Vector similarity search against Neo4j concept_embedding index
3. String normalization (lowercase, expand abbreviations, strip articles)
4. Similarity > 0.85 → auto-resolve to existing slug
5. Similarity 0.6–0.85 → LLM confirmation call (targeted, cheap)
6. Similarity < 0.6 → new concept

Output: Resolved concept list with slug mappings (new vs existing)
3
Pass 2 — Claim Extraction (per-section, parallel)
For each section (semaphore-limited concurrency):
System prompt (generated from schema.yaml):
• Valid claim types + descriptions + required/optional fields
• Valid claim edge types + coupling edge types (ABOUT)
• Instructions: extract assertions, not objects

User prompt:
• Section text
• Full merged concept list from Step 2 (names + slugs + types)
• "For each claim, identify which concepts it is ABOUT"
• "Identify DEPENDS_ON relationships to other claims in this section"

Response format: JSON
{"claims": [{"label": "...", "claim_type": "...", "assumptions": [...], "conclusion": "...", "about_concepts": ["slug1", "slug2"], ...}], "claim_edges": [{"source": "...", "target": "...", "type": "DEPENDS_ON", "role": "existence_guarantee", ...}]}
4
Pass 3 — Cross-Section Edge Enrichment (paper-wide, single call)
One LLM call with the full paper's claims + concepts, focused on:
• Cross-section claim dependencies (Theorem in §4 depends on Lemma in §2)
• Conditional edge attributes: REQUIRES(condition=...), PROVES_VIA(technique=...)
• Strength classification: exact | asymptotic | approximate | existential
• Missing ABOUT edges (claim references concept not caught in per-section pass)

This pass does NOT re-extract — it receives the already-extracted nodes and adds/enriches edges only.
5
Validate & Write to Neo4j
Validate against schema.yaml:
• Every claim has ≥1 ABOUT edge
• Every concept has ≥1 SOURCED_FROM edge
• No unknown edge types or attributes
• DEPENDS_ON edges don't form cycles
• Log warnings/errors per severity in schema

Write to Neo4j (transactional):
• MERGE :Source node (by arxiv_id)
• MERGE :Concept nodes (by slug) — safe-merge: set pipeline-owned fields, skip human-owned if exists
• CREATE :Provenance nodes (always append) → link to Concept + Source
• CREATE :Claim nodes → link SOURCED_FROM Source, ABOUT Concepts
• CREATE claim-to-claim edges with full attribute properties
• Flag concepts with divergent formulations as status: review_needed
LLM Call Budget — Per Paper
Pass Calls Parallelism Context Size
Pass 1 — Concepts N (one per section) Parallel (semaphore) Section text + schema excerpt + 10-20 existing concepts
Dedup confirmations 0–5 (only for 0.6–0.85 similarity) Parallel Two definitions only (very small)
Pass 2 — Claims N (one per section) Parallel (semaphore) Section text + schema excerpt + full concept list
Pass 3 — Enrichment 1 Sequential All extracted nodes + edges (no full text)
Total 2N + 1 + 0–5 dedup For a 10-section paper: ~21–26 calls (vs current ~11)
Embedding computation is separate (local or API) — not counted as LLM calls.
Token cost roughly 2x current pipeline due to the second extraction pass + enrichment.
Schema-Driven Prompt Generation
The prompt generator reads schema.yaml at runtime and produces LLM prompts dynamically.
This means adding a new concept type or edge type is a schema edit, not a code change.
Input: schema.yaml
concept_types:
  structure:
    description: "Named mathematical objects"
    required_fields: [name, formal_spec]
    optional_fields: [components, notation]
  space:
    description: "Sets, domains, topologies"
    ...
→
Output: LLM System Prompt
Extract mathematical concepts from the text.

Valid concept types:
- structure: Named mathematical objects
  Required: name, formal_spec
  Optional: components, notation
- space: Sets, domains, topologies
  Required: name, formal_spec
  ...

Valid concept edge types:
- GENERALIZES: Source is a more general
  form of target. Attributes: condition
- HAS_COMPONENT: Target is a named part
  ...
